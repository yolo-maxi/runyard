Quickstart

This page is the canonical Runyard quickstart — install, run, topology, CLI, MCP, runner pool, security, env vars, and verification, all in one place. The README points here so docs and code never drift.

If you just want to see Runyard work, open the landing page and hit Try it: run hello. Come back here when you want to wire MCP, CLI, API, or a runner pool to a real deployment.

What is Runyard

Runyard is a self-hosted control plane for agent runs. It exposes a capability catalog over MCP, CLI, HTTP API, and a Web Hub; dispatches work to local or remote runners; and stores logs, events, artifacts, approvals, skills, agents, and knowledge centrally. The codebase ships as the runyard package; the legacy SMITHERS_HUB_* environment variables are still honored as fallbacks so existing deployments keep working.

Concepts

A handful of objects describe everything Runyard does. The Hub is the durable source of truth for all of them.

  • Capability. A public, named action an agent or operator can discover and run (for example hello, improve). Capabilities are what show up in the catalog over MCP, CLI, and the Web Hub. In the app and API they are surfaced under workflows.
  • Workflow. The implementation behind a capability — the Smithers steps and agents that actually do the work. Most users interact with capabilities; workflows are the editable internals.
  • Run. One durable execution of a capability. A run moves through states: queued → running → succeeded/failed/cancelled, and pauses in waiting_approval when a human checkpoint is required. Every run keeps its logs, events, outputs, artifacts, and a unified timeline.
  • Runner. A registered execution environment — a process on a VPS or laptop that polls the Hub, claims matching queued runs, executes them, streams events, and uploads artifacts. Runners are disposable; the Hub holds the state.
  • Approval. A human checkpoint. A run that needs sign-off enters waiting_approval and is resolved (approve / reject / request changes) from Web, API, CLI, MCP, or Telegram — recorded once no matter the surface.
  • Artifact. A persistent file output attached to a run, stored on the Hub under data/artifacts/runs/<run-id>/ and downloadable from the run detail page or API.
  • Agent. A reusable role/profile (instructions, model, tools) a workflow step can adopt.
  • Skill. A reusable instruction/tooling package a workflow or agent can pull in.
  • Knowledge resource. Shared company context agents can search while they work.
  • Access token. The single auth primitive. One bearer token authenticates Web, API, CLI, MCP, and runners; scopes narrow what a token can do.

Install

Requirements: Node.js 22.5+ (the server uses --experimental-sqlite), pnpm, and a writable data/ directory.

git clone https://github.com/yolo-maxi/runyard
cd runyard
pnpm install
pnpm build   # builds the web app and copies self-hosted vendor CSS

Public clients can also install the CLI from a running hub with bash <(curl -fsSL https://hub.example.com/install.sh). hub.example.com is just an example domain — use whatever you host on.

Run locally

PORT=43117 BASE_URL=http://127.0.0.1:43117 pnpm start
# server writes data/bootstrap-token.txt on first boot
open http://127.0.0.1:43117/app

Topology options

Runyard is built for one private Hub per company or org. The Hub owns durable state; runners are disposable workers that poll over HTTPS. You can adapt the deployment shape to your threat model and hardware.

  • Small/local. Hub and one runner on the same machine. Best for development or a solo operator. Keep runner capacity low.
  • Private org. Hub on a small VPS with TLS reverse proxy, SQLite/artifacts on local disk, runners on separate machines.
  • Dedicated runner pool. One or more worker VPSes advertise tags and capacity. The Hub queue stays centralized; runners do not access SQLite directly.
  • Public docs + private Hub. Serve setup docs as a static public site, while `/app`, API, MCP, runner tokens, SQLite, and artifacts stay on the private Hub.
  • Our deployment. runyard.repo.box is static docs. hub.repo.box is the live token-protected Hub. Build/agent work runs on a separate worker host, not on the serving box.

Deploy

  • Node + SQLite single-process server. No external database.
  • Artifacts on local disk under data/artifacts/. Back up by copying the data/ directory.
  • Bearer-token authentication only — no SaaS user database.
  • One private deployment per company/org. Each gets its own domain & bootstrap token.
  • Reverse-proxy with TLS in front (Caddy, nginx, Traefik). The Hub does not terminate TLS itself.
  • Runners may live on the same box or on remote VPSes, Linux workstations, or macOS laptops.

Auth tokens

Runyard uses long-lived access tokens. On first boot the server writes data/bootstrap-token.txt — that becomes the operator key for Web login, CLI, MCP, API, and runner registration. Issue additional tokens from the Connect tab inside the Hub.

CLI

runyard login --url https://hub.example.com --token shub_...
runyard menu
runyard capabilities
runyard capability hello
runyard run hello --where local --input '{"topic":"Prepare a rollout spec"}'
runyard run hello --where remote --input '{"topic":"Prepare a rollout spec"}'
runyard runs
runyard tail run_... --once
runyard runners
runyard approvals
runyard approve appr_...

Use --where local for a laptop/workstation runner and --where remote for the VPS runner pool. The Hub records the run, logs, outputs, artifacts, and a unified timeline either way. runyard tail <run-id> streams that timeline as NDJSON.

Multiple orgs work like git remotes:

runyard login --remote acme --url https://acme-hub.example
runyard mcp install --all --remote acme   # installs as runyard-acme
runyard remote use acme
runyard remotes

MCP config

{
  "command": "runyard-mcp",
  "env": {
    "RUNYARD_HUB_URL": "https://hub.example.com",
    "RUNYARD_HUB_TOKEN": "shub_..."
  }
}

From a running Hub, runyard mcp install --all writes this config into every detected agent — Claude Code/Desktop, Codex, Cursor, Windsurf, Gemini, VS Code.

MCP agents should call get_menu, choose executionMode: "local" or "remote" on run_capability, then fetch status, logs, artifacts, and timeline entries from the Hub tools.

HTTP API

Everything the Web Hub, CLI, and MCP server do runs over the same JSON API under /api. Authenticate every request with Authorization: Bearer shub_.... The machine-readable contract lives at /openapi.json and the agent-facing summary at /llms.txt; both are generated from the live Hub so they never drift.

# Discovery (no drift — same source as get_menu)
GET  /api/menu                       # tools, capabilities, execution modes
GET  /api/capabilities               # catalog
GET  /api/capabilities/:id           # one capability + input schema

# Runs
POST /api/capabilities/:id/run       # body {input, executionMode: "local"|"remote", responseEndpoint?}
GET  /api/runs                       # list (filter by status, q, capability)
GET  /api/runs/:id                   # status, outputs, responseEndpoints[]
GET  /api/runs/:id/timeline          # unified ascending NDJSON-style timeline (since, limit)
GET  /api/runs/:id/logs              # log lines
GET  /api/runs/:id/artifacts         # artifacts;  GET /api/artifacts/:id/download
POST /api/runs/:id/rerun             # re-queue with the same (or edited) input
POST /api/runs/:id/cancel

# Approvals
GET  /api/approvals
POST /api/approvals/:id/approve | /reject | /request-changes

# Catalog editing & ops (admin scope)
GET/POST/PATCH  /api/agents | /api/skills | /api/knowledge
GET/POST        /api/tokens          # issue scoped access tokens
GET             /api/audit           # audit log

# Assistant
GET  /api/chat/status                # provider + whether it's configured
POST /api/chat                       # { messages, context } → reply (+ optional action buttons)

# Liveness (unauthenticated)
GET  /healthz  /readyz  /api/version

Most run, capability, artifact, and approval responses include a deepLink field — a stable /app#… URL you can hand an operator. A run can carry an optional responseEndpoint ({type: "http"|"telegram", config}) so the Hub delivers the terminal-state reply when the run finishes; polling /api/runs/:id always stays canonical and delivery state shows up under responseEndpoints[].

Runner pool

Runners are stateless processes that poll the Hub for queued work. Each runner advertises a tag set and an optional capacity:

RUNYARD_HUB_URL=https://hub.example.com \
RUNYARD_HUB_TOKEN=shub_... \
SMITHERS_RUNNER_NAME=hetzner-vps-runner \
SMITHERS_RUNNER_LOCATION=vps \
SMITHERS_RUNNER_TAGS=linux,node,git,shell,web,smithers \
SMITHERS_RUNNER_CONCURRENCY=4 \
runyard-runner
  • Tag match. The Hub only schedules a run on a runner that advertises every required tag.
  • Location match. Local runs target runners tagged local; remote runs target runners tagged vps or remote.
  • Capacity. A pool runner with SMITHERS_RUNNER_CONCURRENCY=4 can hold four concurrent slots; the Web Hub shows the slot grid live.
  • No double-claim. Two runners polling for the same queued run can never both pick it up — the Hub atomically assigns one and rejects the other.
  • Disposable. Restart freely; the Hub keeps the durable record.

Assistant

Every Hub page has a floating Assistant — an in-app support copilot that answers questions about Runyard and what's on your screen. Click the chat button in /app to open it.

  • Answer-first, never surprise-runs. The Assistant is a chat agent. It explains what happened before proposing anything. App-changing actions (navigating, triggering a run, calling the API) only execute when you click a confirmation button it offers — a question stays a question.
  • Context-aware. It always receives your current route, and the server resolves a redacted, read-only snapshot of what you're actually looking at — the failing run's recent events, the runner pool's free slots, pending approvals — so it can answer "why did this fail?" from real state, not guesses. Secrets and token-shaped strings are stripped before the model ever sees them.
  • Runner-backed by default. Out of the box the provider is the Hub runner pool: /api/chat queues a tiny internal runyard-support-agent workflow and your subscribed on-runner CLI agent answers it — no extra API key needed. Direct HTTP providers are opt-in for installs that prefer them: set RUNYARD_HUB_SUPPORT_AGENT_PROVIDER=anthropic (with ANTHROPIC_API_KEY) or =openai (with OPENAI_API_KEY).
  • Configurable. Disable it with RUNYARD_HUB_SUPPORT_AGENT_ENABLED=0. Tune with RUNYARD_HUB_SUPPORT_AGENT_MODEL, _TIMEOUT_MS, and _MAX_OUTPUT_TOKENS. GET /api/chat/status reports the resolved provider and whether it's configured.

Telegram approvals

Approval notifications are designed for a private operator DM. Configure TELEGRAM_BOT_TOKEN plus TELEGRAM_APPROVAL_CHAT_ID or SMITHERS_TELEGRAM_APPROVAL_CHAT_ID. Legacy TELEGRAM_CHAT_ID/TELEGRAM_THREAD_ID remains a fallback when no private approval chat is set. Approve/reject/request-changes from Web, API, CLI, MCP, or the Telegram buttons; the decision is recorded once in the Hub regardless of surface.

Environment variables

Required for production:

  • BASE_URL — the public origin (e.g. https://hub.example.com).
  • PORT — port the server binds to.
  • RUNYARD_HUB_SESSION_SECRET — long random string. Sessions and CSRF derive from it.
  • RUNYARD_HUB_BOOTSTRAP_TOKEN or read access to data/bootstrap-token.txt.

Optional:

  • TELEGRAM_BOT_TOKEN, TELEGRAM_APPROVAL_CHAT_ID (or SMITHERS_TELEGRAM_APPROVAL_CHAT_ID) for approval DMs.
  • TELEGRAM_CHAT_ID / TELEGRAM_THREAD_ID as a backward-compatible fallback.
  • RUNYARD_HUB_INSTANCE_NAME to label a deployment in /api/version.
  • RUNYARD_HUB_SUPPORT_AGENT_ENABLED (default on), _PROVIDER (runner|anthropic|openai), _MODEL, _TIMEOUT_MS, _MAX_OUTPUT_TOKENS — the in-app Assistant. HTTP providers also read ANTHROPIC_API_KEY / OPENAI_API_KEY.

Runner-side variables (used by runyard-runner):

  • RUNYARD_HUB_URL, RUNYARD_HUB_TOKEN — connection & auth.
  • SMITHERS_RUNNER_NAME, SMITHERS_RUNNER_LOCATION, SMITHERS_RUNNER_TAGS, SMITHERS_RUNNER_CONCURRENCY.
  • SMITHERS_WORKSPACE — directory the runner uses as scratch space for Smithers workflows.
  • IMPROVE_REPO_DIR or GATED_REPO_DIR — default git repo for the Improve workflow.
  • IMPROVE_ALLOWED_REPO_ROOTS — comma-separated or JSON array of extra absolute roots that Improve may edit via repoDir.
  • IMPROVE_REPO_MAP, IMPROVE_PROJECT_MAP — JSON objects mapping friendly repo/project input keys to absolute runner-local git repo paths.

For Improve, repoDir selects the allowlisted runner-local repository to inspect, test, commit, push, and optionally deploy. The Hub still stores the run status, logs, outputs, and artifacts.

IMPROVE_ALLOWED_REPO_ROOTS='["/srv/repos"]'
IMPROVE_REPO_MAP='{"website":"/srv/repos/website"}'
runyard run improve --where remote --input '{"target":"pricing page","repo":"website"}'

Workflow hardening

Runyard is designed to move workflows from agentic exploration toward deterministic code. The operating philosophy follows the process order: question requirements, delete unnecessary steps, simplify, accelerate, then automate. In Runyard terms, automation means moving from agent judgment to machine-checkable scripts and code where the work has become repeatable.

Agents often write shell scripts, one-off commands, data extractors, and verification snippets while completing work. Runyard treats those as process discoveries. The long-term goal is to extract that knowledge, turn it into named workflow steps, and replace agent discretion with tested code when the output contract is stable.

  • L0 raw agentic. A prompt asks an agent to figure the work out. High variance is expected.
  • L1 constrained agentic. Inputs, outputs, and evals are explicit, but the step is still mostly agent judgment.
  • L2 checkpointed. The step writes a resumable checkpoint and recovery instructions.
  • L3 script-backed. The common path is a script; an agent supervises exceptions.
  • L4 deterministic code. The step is code with tests and typed inputs/outputs.
  • L5 automated machine step. The step runs without an agent unless anomaly detection asks for one.

The nightly optimizer loop should replay representative workflow fixtures, diff outputs, measure variance/failure/cost, and open improvements that delete, split, simplify, or harden steps. Creative and taste-heavy work may remain agentic; engineering plumbing should harden aggressively.

See specs/workflow-hardening-and-optimizer.md for the durable product spec.

Security model

  • Single auth primitive. Every surface (Web, API, CLI, MCP, runner) authenticates with a Hub access token.
  • Private deployment. Runyard is meant for one org per deployment behind a domain you control. There is no multi-tenant SaaS database.
  • Static public docs. If you publish a marketing/setup site, make it static. Do not expose `/app`, access tokens, SQLite files, artifacts, or environment files.
  • Bootstrap token. The first token is written to data/bootstrap-token.txt; rotate it via the Connect tab once you are logged in.
  • Scopes. Access tokens carry scopes — api, mcp, runner, and admin (issue them from the Connect tab) — so a pool host or approver gets only the access it needs. admin is required for token, audit, and workflow-endpoint management and implies the others; the bootstrap token is full admin. Approval-only sessions (e.g. Telegram web-app login) hold just the approvals scope.
  • Permissive runners. Runners execute work on behalf of authenticated requests. Treat the runner host as if it has the privileges of any operator who can submit a capability run; isolate sensitive runners with restricted tag sets.
  • Headers. The server sets strict content-security-policy, x-content-type-options, referrer-policy, and strict-transport-security (when served over HTTPS).
  • Rate limiting. Login and general API buckets are enforced to discourage credential stuffing and scraping.

Manual checks after install:

  1. Log in to /app with the bootstrap token.
  2. Run Hello with --where local or --where remote.
  3. Start a runner and confirm the run succeeds.
  4. Open the run detail page and download the generated artifact.
  5. Run Implement; approve it via Web or MCP; confirm it queues and executes.
  6. Configure a local MCP client with runyard-mcp and call list_capabilities.

Open source & contributions

  • The repo is structured for open-source release: bin/, src/, public/, specs/, workflow-templates/, tests/.
  • Specs live under specs/. Read them before changing product behaviour — they describe intent, expectations, decisions, and acceptance checks.
  • The gated pipeline runs pnpm test, git diff --check, and smoke-checks /healthz, /app, /docs, /llms.txt before deploy.
  • File issues and PRs against the public Runyard repo. Keep private hostnames out of code and use hub.example.com in docs.

Deep links

Every console page, run, workflow, agent, skill, knowledge entry, run artifact, and approval has a stable URL — paste it into chat or docs and it survives reload. Each card and detail view shows a 🔗 button to copy that link to your clipboard. Examples:

/app#runs                                 # all runs (home)
/app#runs/<id>                            # one run's detail
/app#runs/<id>/logs                       # jump to the log timeline
/app#runs/<id>/artifacts                  # jump to artifacts
/app#runs/<id>/artifacts/<artifact-id>    # one artifact inside its run
/app#workflows/<slug>                     # workflow detail (description, agents, runs)
/app#workflows/<slug>/run                 # workflow detail with the Run form open
/app#workflows/<slug>/runs                # latest runs for this workflow
/app#workflows/<slug>/edit                # workflow detail with the Edit modal open
/app#agents/agents/<slug>                 # a specific agent
/app#agents/skills/<slug>                 # a specific skill
/app#agents/knowledge/<slug>              # a knowledge entry
/app#approvals/<id>                       # an approval
/app#tokens | #runners | #audit | #connect | #settings

The same URLs come back as deepLink fields on most API responses (runs, capabilities, artifacts, approvals) so MCP/CLI clients can hand operators a clickable link with zero extra work.
