How it works

One hub. A pool of runners. Full visibility end to end.

RunYard is a hub — the control plane: API, web UI, durable datastore, and run queue — plus one or more delegated runners that execute the actual work. Runners can live anywhere: a VPS, your laptop, Kubernetes, or a confidential VM.

Trigger & approve

Agents & people

  • MCP server for AI clients
  • CLI, HTTP API & web UI
  • Approve from Web, CLI, API, Telegram
Control plane

The hub

  • Durable datastore & run queue
  • Live event tail & streaming logs
  • Approval gates & full run history
Execute anywhere

Delegated runners

  • VPS, laptop, k8s, or confidential VM
  • One pool, route work by tags
  • Disposable workers, durable record

Under the hood the execution engine is Smithers (smithers-orchestrator) — RunYard is the control plane around it.

Why RunYard

Everything you need to run agents in production — that you actually control.

Durable & crash-safe

Runs are persisted as they progress. Restart the hub or a runner and work resumes from where it left off — no lost progress, no orphaned jobs.

Full visibility

A live streaming console and event tail for every run — like docker logs -f for an agent. See each step, tool call, and artifact as it happens.

Human-in-the-loop

Gate sensitive steps behind approvals. Review and approve or reject from the Web UI, CLI, API, MCP, or Telegram — without re-implementing gates per surface.

Runs on your infra

Self-hosted by design. Code, data, and credentials stay on machines you control — bearer-token auth, per-deployment data dir, no SaaS dependency.

One pool, many machines

Bring more capacity online with one process per host. Runners join a single pool and you route work by tags — linux, gpu, git, your call.

Deploy anywhere

Ships as Docker images (GHCR) plus docker-compose — a portable base with per-target overlays, including confidential VMs (dstack / Intel TDX) for hardware-attested isolation.
