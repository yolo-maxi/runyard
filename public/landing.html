Built for agents to drive — and for you to inspect.

Runyard treats AI agents as first-class operators. The capability catalog, runs, logs, artifacts, and approvals are one product exposed over four surfaces — same data, same token, no drift between what your agent sees and what you see.

The whole surface is machine-readable: openapi.json and llms.txt are generated from the live hub, so they never drift from the code.

Human-in-the-loop

Humans are slow. Runs should wait, not die.

Gate the step that matters — pushing to main, spending money, touching prod — and let everything before it run unattended.

  • Held means held. A run waiting on a person is parked in waiting_approval — it is never age-reaped or deadline-failed while it waits. Approve ten hours later and it resumes at the exact step it paused on.
  • Deadlines are explicit, never implicit. Want a time limit? Attach a timer with a fallback decision and the hub applies it on your behalf, fully audited. A timer without a fallback only escalates — it never decides for you.
  • Answer from wherever you are. Web, CLI, API, MCP, or a Telegram DM — one decision, recorded once, with the who/when/where in the run's permanent timeline.

How it works

One hub. Any number of runners. All of it yours.

The hub is the control plane — API, web UI, run queue, and a durable datastore. Runners are disposable workers that claim matching work and stream everything back. Kill any of them; the record survives.

Why Runyard

Production manners for agent workloads.

Durable & crash-safe

Runs persist as they progress. Restart the hub or a runner and work resumes where it left off — no lost progress, no orphaned jobs.

Watch every step

A live streaming console and unified timeline for every run — each step, tool call, log line, and artifact as it happens, kept forever.

Approvals that hold

Sensitive steps pause the run and wait for a human — from any surface. Late answers resume work; they don't restart it.

Your infra, your data

Self-hosted by design: bearer-token auth with scopes, per-deployment data directory, artifacts on your disk. No SaaS in the loop.

One pool, many machines

One process per host joins the pool and advertises tags and capacity. The hub routes work to matching runners and never double-assigns a run.

Deploy anywhere

Docker images on GHCR plus a portable compose base with per-target overlays — including confidential VMs (dstack / Intel TDX) for hardware-attested isolation.

A hub of your own, before the coffee's done.

Clone, build, start. First boot writes your operator token; from there the web hub, CLI, API, and MCP server are all live on your machine.
