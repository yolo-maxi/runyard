Self-hosted control plane for agent runs

Run your first capability in one click

A capability is a team-defined action — a workflow with inputs, outputs, runner requirements, and an approval policy — that agents discover and run through Runyard. Pick one, hit Try, and watch logs stream while artifacts land on the Hub.

Try it: run hello Quickstart

Sample run uses the default single-host topology. No topology decision required up front — you can wire MCP, CLI, API, or a runner pool from Connect after your first successful run.

$ smithers-hub run hello --where local
→ queued run_01H9 capability=hello runner=local step 1/3 prepare workspace ✓ step 2/3 invoke agent (claude) … log: collected 3 sources, drafting summary step 3/3 write artifact hello-output.md ✓ artifact hello-output.md (2.1 KB) ★ run succeeded · open /app#connect to wire your agents $ _
Advanced: install MCP, CLI, API, or a runner pool

Most operators don't need any of this until after their first successful run. Each card below mirrors the snippets shown in /app#connect; bin names match the current build.

MCP

Wire every detected AI client to this Hub.

smithers-hub mcp install --all
# auto-detects Claude Code/Desktop, Codex, Cursor, Windsurf, Gemini, VS Code

CLI

Authenticate, then run the next-action menu.

smithers-hub login --url https://hub.example.com
smithers-hub menu                # try: smithers-hub run hello
smithers-hub menu --all          # full catalog

HTTP API

Bearer-token API mirrors every CLI/MCP action.

curl -H "authorization: Bearer $TOKEN" \
  https://hub.example.com/api/menu
curl https://hub.example.com/llms.txt

Runner pool

Bring more capacity online with one process per host.

SMITHERS_HUB_URL=https://hub.example.com \
SMITHERS_HUB_TOKEN=shub_... \
SMITHERS_RUNNER_TAGS=linux,node,git,shell,web,smithers \
smithers-hub-runner

What Runyard gives you

Capability Catalog

Agents see a clean menu of team-defined actions with input/output schemas, required runner tags, skills, and approval policy.

Local and Remote Runners

Run work on a VPS, laptop, or workstation; the Hub keeps the durable record and the runners stay disposable.

Unified Approvals

Approve through Web, API, CLI, Telegram, or MCP without re-implementing workflow gates per surface.

Shared Company Memory

Editable skills, agents, and knowledge entries become reusable infrastructure rather than scattered prompts.

Open agent surface

Stable HTTP API, /openapi.json, /llms.txt (rendered from get_menu), and an MCP server keep agents and humans on the same surface.

Private by default

Bearer-token auth, per-deployment data dir, no SaaS dependency. Bring your own domain and TLS terminator.

Workflow hardening

Runyard treats agent workflows as processes to improve, not just prompts to rerun. Inspired by the process order “question, delete, simplify, accelerate, automate,” workflows start agentic, then harden into smaller scripted and coded steps wherever outcomes become repeatable.

Discover with agents

Early runs keep agents close to the work so they can explore, write throwaway scripts, learn edge cases, and produce the first useful deliverables.

Extract the repeatable bits

Runyard keeps the scripts, commands, evals, and artifacts agents create, then turns stable patterns into explicit workflow steps with inputs, outputs, and checks.

Compile toward code

Nightly optimizer runs replay workflows, diff outcomes, delete unnecessary steps, and propose deterministic code paths for the parts that no longer need agent judgment.

Runyard is the public name for the project formerly called Smithers Hub. Package, bin, and env names keep the smithers-hub prefix so existing deployments and tokens keep working.
