# RunYard v0.11: Hub-native supervision

Status: planned for the post-v0.10 reliability train.

Owner: Fable/Codex implementation lane, reviewed by Ocean.

## Why this exists

The earlier direction was to put normal user-startable workflows behind the
`run-smithers` wrapper by default. Live production evidence showed that this
made failures noisier and less recoverable:

- A child workflow preflight failure became both a child failure and a parent
  `run-smithers` failure.
- The parent workflow often had less state than the Hub already had.
- Human approval state could drift between Smithers, the Hub, and the wrapper.
- Retries could happen from inside another workflow after side effects.
- The run list showed doubled failures for one operator-visible problem.

The correction is not "remove supervision." The correction is to move the
supervision decision closer to the Hub scheduler/state machine, and keep
workflow execution itself direct and understandable.

## Roadmap slot

This is the v0.11 reliability milestone. It supersedes the old
`specs/codex-goal-supervise-all-ui-runs.md` direction.

Required ordering before v0.11 starts:

- Current `main` must include the prod hotfix `ca631ae` or later.
- The approval-timeout semantics branch should merge first, because v0.11
  depends on the invariant that late humans do not create terminal failures.
- Active v0.8/v0.9/v0.10 Fable branches must rebase or merge current `main`
  before they are merged, so the hotfix and this roadmap correction are not
  lost.

Recommended release train shape:

- v0.8: finish child-env and runner hygiene.
- v0.9: finish endpoint/harness selection and operational install pieces.
- v0.10: stabilize the current product train and release evidence workflow.
- v0.11: make supervision Hub-native and remove default wrapper behavior.

## Objective

Make normal user/API/UI-started workflows run directly by default, with clear
preflight, gates, isolated worktrees where needed, promotion, approvals, and
terminal status. Move retry/resume/escalation policy into Hub-native state and
reconcile loops.

`run-smithers` remains available, but only as an explicit/manual workflow until
it is slimmed into a thin executor or recipe over Hub primitives.

## Non-goals

- Do not delete `run-smithers` in this pass.
- Do not reintroduce default wrapping through another hidden capability layer.
- Do not expand runner host topology. The current assumption remains one host,
  shared local workspace, multiple runner processes.
- Do not weaken preflight or promotion gates to make direct execution "easier."
- Do not change marketplace/payment semantics.

## Phase 0: stop making the problem worse

Implement before the next production cut if possible.

- Disable default supervision wrapping for normal user-startable workflows.
- Preserve explicit `run-smithers` launches for manual experiments and support.
- Update capability metadata/tests so `improve`, `implement-change-gated`,
  `idea-to-product`, `app-skinner`, and similar workflows do not create a
  parent wrapper by default.
- Keep internal bypass protection so a manual `run-smithers` child cannot
  recursively wrap itself.
- Update the UI/API expectation: a user-started run should normally be the run
  that executes, not a wrapper that starts another run.

Phase 0 acceptance criteria:

- Creating an `improve` run through the API creates one executable run, not a
  `run-smithers` parent plus child.
- Creating `implement-change-gated` and `idea-to-product` runs follows the same
  direct default.
- Explicitly starting `run-smithers` still works and records child lineage.
- Run list/detail no longer shows doubled failures for one direct preflight
  failure.
- Tests prove default direct execution and explicit wrapper behavior.

## Phase 1: Hub-owned recovery decisions

Build on `docs/design/hub-supervisor.md` and the shipped `src/hubSupervisor.js`
work instead of starting from scratch.

- Treat the Hub reaper/reconcile path as the supervision owner.
- Classify orphaned, stalled, and recoverable failed runs from Hub state.
- Requeue resumable runs through existing scheduler primitives.
- Escalate to operator approval when caps, repeated fingerprints, missing
  checkpoint, unsafe side effects, or ambiguity make automatic recovery unsafe.
- Record lineage for every resume/escalation decision.
- Keep decision logic pure where possible so unit tests can cover the state
  machine without booting the full server.

Phase 1 acceptance criteria:

- Runner-offline with resumable checkpoint requeues instead of terminal-failing.
- Requeue uses compare-and-swap protection and cannot double-dispatch.
- Same fingerprint without progress hits a loop breaker and escalates.
- Cancellation remains terminal and is never auto-resumed.
- Approval-held runs do not fail from stall or max-runtime age alone.

## Phase 2: repair policy moves out of the wrapper

- Reuse the existing deterministic-code-bug classifier and repair budget.
- When Hub state says a repair is appropriate, dispatch a bounded repair run
  with explicit purpose and lineage.
- Resume only after the repair succeeds and the resume target is still valid.
- If repair fails or budget is exhausted, escalate instead of looping.

Phase 2 acceptance criteria:

- One deterministic workflow/template bug can trigger one repair attempt.
- Repair attempt is visible in run lineage.
- Repaired workflow is synced where runners execute it.
- Repair failure escalates with useful evidence.
- No repair path can run after an irreversible side effect unless the workflow
  explicitly marks that step safe to resume.

## Phase 3: slim `run-smithers`

- Remove policy decisions from `run-smithers` that duplicate Hub state.
- Keep it as an explicit orchestration/executor recipe if it still adds value.
- Do not use it as the default wrapper for ordinary UI/API-created runs.
- Make docs/UI name it as advanced/manual, not the universal safety layer.

Phase 3 acceptance criteria:

- `run-smithers` no longer claims to be the default supervision layer.
- README and specs describe Hub-native supervision as the source of truth.
- Manual wrapper smoke still works or is deliberately marked experimental.

## Verification before merge

- Unit and integration tests from all touched areas pass.
- GitHub CI is green.
- Any active Fable branches are rebased/merged on top of current main.
- The old `supervise-all-ui-runs` direction is not referenced as active.
- The project brief records the new v0.11 slot and execution branch.

## Production deployment gate

Do not deploy v0.11 directly after merge without a canary:

- Run a direct `improve` smoke with explicit repo/project.
- Run a direct non-mutating capability smoke.
- Run explicit `run-smithers` smoke and confirm it is manual/advanced.
- Confirm failed direct preflight produces one failed run, not a parent and
  child failure pair.
- Verify `/healthz`, `/version`, `/app`, `/llms.txt`, `/openapi.json`, runner
  heartbeats, support runner, and Hub auth self-checks.
