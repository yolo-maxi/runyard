## Context — read first
RunYard is being packaged as a **product that third parties self-host**. The maintainer (us) has
**zero control and zero access** to an operator's box. The box must check for updates and let the
**operator decide** whether to apply — no phone-home, no remote control, ever. The only external
network call permitted is an **outbound** read of the public GitHub Releases API.

## The shape: CHECK ≠ APPLY
- **Check** = passive, automatic, read-only. Poll GitHub Releases, compare latest tag to running
  version, surface "update available". NEVER installs anything.
- **Apply** = operator-initiated only (UI button or `runyard update`). Self-healing.

## Deliverables

### 1. CI release workflow (`.github/workflows/release.yml`)
- On push of a tag matching `v*`: checkout, setup node 22 + pnpm, `pnpm install --frozen-lockfile`,
  `pnpm test`. **Only if green**, create/publish the GitHub Release for that tag (use
  `softprops/action-gh-release` or `gh release create`). A red test run must NOT publish a release.
- Also run `pnpm test` on PRs / pushes to `main` as a gate (can be the same workflow or a second job).

### 2. Version + update-check
- A canonical version source: read `package.json` `version` (and bump it to a real semver if it is a
  placeholder). Expose **`GET /version`** → `{ version, gitTag, gitCommit }` (no auth needed — it's
  just the running version; do NOT leak anything sensitive).
- An update-check module that calls `https://api.github.com/repos/<owner>/<repo>/releases/latest`
  (owner/repo configurable via env, default the RunYard repo), parses `tag_name`, semver-compares to
  the running version, and caches the result (don't hammer the API — cache ~1h, respect rate limits,
  handle network failure gracefully = "unknown", never crash).
- Config in `env.js`: `UPDATE_CHECK_ENABLED` (**default ON** — passive + outbound-only is low risk;
  must be toggleable to off), `UPDATE_CHECK_INTERVAL_MS`, `GITHUB_REPO` (owner/repo).
- UI: an "update available → vX.Y.Z" badge in the hub UI, modeled on the auth-health strip. Shows
  current vs latest; only visible to admin scope. When up-to-date, show nothing or a subtle "latest".

### 3. `runyard update` — operator-triggered apply, self-healing
Build as a script (`bin/runyard` CLI subcommand and/or `scripts/runyard-update.sh`) AND wire an
admin-only UI/endpoint trigger (`requireScopes("admin")`). The apply sequence:
1. Resolve target tag (latest, or an explicit `vX.Y.Z` arg).
2. **Record the current tag/commit as last-known-good** (persist to a file under `dataDir`, e.g.
   `dataDir/.last-good-release`).
3. **Drain runners** before swapping: set a `.drain` flag the runner checks **before claiming new
   work** (add this check in `smithers-runner.js`); the runner finishes in-flight work and stops
   claiming. Poll `active_runs` / `current_run_id` until 0, with a **bounded grace window** (default
   ~45 min, configurable). If it does not drain in time: **abort the update**, clear the drain flag,
   record an alert, exit non-zero. Default is ALWAYS "wait, don't kill in-flight work."
4. `git fetch --tags`, checkout the target tag.
5. `pnpm install --frozen-lockfile` **only if the lockfile changed** (diff old vs new lockfile).
6. Restart the unit(s). Poll **`/healthz`** for up to ~30s.
7. **On healthcheck failure → AUTO-ROLLBACK:** checkout last-known-good tag, reinstall deps if
   needed, restart, re-verify health, and write an alert "update failed, rolled back to vX". A bad
   release must never be able to brick the hub.
8. Record outcome (success/failed-rolled-back) in `_smithers_alerts` so the UI badge can show it.
   Optional `.env` `UPDATE_NOTIFY_WEBHOOK` — if set, POST the outcome to that generic webhook (the
   operator's own Slack/Discord). If unset, UI alert + logs only. **No phone-home to us, ever.**
- **Re-exec safety:** the updater swaps code while running. Keep the update script dead-simple /
  self-contained so it isn't broken by the swap, or have it copy itself to a temp path and re-exec
  from there before touching the working tree. Document the chosen approach in a comment.
- **Hub vs runner asymmetry:** runners MUST drain (mid-run restart destroys work). The hub may
  restart more freely (Smithers runs are DB-durable and resume) — still prefer to restart when no
  runs are `running`, else defer. Encode this difference.

### 4. Installer (`install.sh`, runnable via `curl … | bash`)
One-command fresh-box bootstrap: check/install node 22 + pnpm, clone the repo at the latest release
tag, generate `SECRETS_ENC_KEY`, write a starter `.env` (prompt or sane defaults; ports, repo,
`REAUTH_ENABLED`, update-check config), create the `data/` dir, install + enable the systemd unit(s)
(reuse `deploy/*.service`), print the Caddy reverse-proxy snippet the operator should add (do not
assume Caddy is present — print instructions), and install the update mechanism (timer/cron for the
passive *check* only — NOT auto-apply). Idempotent: re-running must not clobber an existing `.env`,
`data/`, or `SECRETS_ENC_KEY`. Be defensive; this runs on machines we never see.

## Engineering rules (non-negotiable)
- **Migrations stay additive & backward-compatible** (ADD column/table, never rename/drop) so a
  rollback to older code boots cleanly against a newer-migrated DB. Follow the existing `db.js`
  pattern exactly.
- **Never touch `data/`** during update except to read/write the last-good marker. Secrets + DB
  survive every update and rollback.
- No phone-home. Outbound GitHub Releases read is the only external call. No telemetry, no callbacks
  to any maintainer-controlled endpoint.
- Secrets never logged. Don't print `SECRETS_ENC_KEY` or any secret to stdout/logs.
