North Star

RunYard is not trying to be a pretty agent dashboard. It should be the installable control plane you trust when long-running AI work touches real repositories, deployments, budgets, approvals, and operators.

Durable

Runs survive reality

Every run keeps lineage, inputs, events, artifacts, approvals, outcomes, and recovery context.

Inspectable

No hidden magic

Operators can see workflow source, graph shape, runner state, failure class, and why approval is requested.

Replay-aware

Retries know side effects

Checkpoint presence alone is not enough. Replay safety must be explicit and enforced.

Bounded

Power comes with fences

Approval, budget, provider, runner, sandbox, and learning boundaries become product contracts.

Current Architecture

The Hub is the source of truth. Runners execute work. Smithers provides workflow orchestration. RunYard adds product contracts, approval paths, durable traces, operator UX, and recovery policy.

1
Agent or human starts a capability Via Web, CLI, MCP, API, or a supervising workflow such as run-smithers.
2
Hub records a durable run Inputs, capability, workflow source, status, runner requirements, and approvals are tracked centrally.
3
Runner claims matching work Runner tags and auth health determine what can run where. Future versions should add stronger sandbox policy.
4
Smithers executes workflow steps Agentic, scripted, deterministic, and approval-gated steps stream events and artifacts back to the Hub.
5
Finalization is explicit Commit, push, promote, deploy, notify, and mutate operations should happen behind prechecks and replay-safe fences.

Hub

Express/Node service backed by SQLite today. Stores runs, events, artifacts, approvals, tokens, runners, capabilities, agents, skills, and knowledge.

Runner

Polls the Hub, claims matching queued runs, launches Smithers or local execution, streams logs/events, uploads artifacts, and reports outcome.

Smithers

Workflow substrate for durable multi-step execution, checkpoints, approvals, and resumable orchestration.

run-smithers

Supervising wrapper that records child lineage, classifies repeated failures, and asks for human recovery after repeated fingerprints.

Approvals

Human checkpoints are stored in Hub and resolved through Web, API, CLI, MCP, Telegram, and future Slack/webhook endpoints.

Promotion

Recent work isolates implementation in branches/worktrees and uses explicit merge-to-main promotion with gates and cleanup.

Known Failure Points

Reliability has been the weak spot. These are the failure classes that need product-level treatment before RunYard can be trusted near serious deployments.

Unsafe replay

Failure after mutation

A run can commit or push successfully, then fail during deploy or reporting. If retry logic only sees a checkpoint, it can replay work and duplicate side effects.

Terminal propagation

Local failure hidden by stale Hub state

If the runner fails to report terminal status, the Hub may later classify the run as stalled instead of preserving the real root cause.

Stale workflow copies

Runner workspace drift

Workflow-template changes in source can be bypassed if live runner workspaces execute stale copied workflows.

Approval stalls

Waiting is not dead

Approvals must be excluded from generic stall reaping and surfaced clearly across all approval endpoints.

Prompt transport

Large input E2BIG failures

Large prompts or structured inputs must move through stdin/files, not giant argv strings.

Provider ambiguity

Auth, quota, rate limit, timeout

Provider failures need typed classification and policy, not a generic failed status that encourages blind retry.

Roadmap Toward v1.0

The roadmap works backward from operator trust. Each milestone gets a feature theme and a matching refactor theme.

0.3

Workflow-as-code foundation

Workflow code model, source viewer/editor, validation, test harness, precheck interface, and CVM-friendly authoring.

0.4

CVM install and runner deployment

Digest-pinned UpgradeApp path, compose/dstack docs, runner capability model, and first sandbox contract.

0.5

Reliability and modular approvals

Approval endpoint abstraction, Telegram hardening, Slack/webhook proof, side-effect replay fences, and better failure classes.

0.6

Token, budget, and provider control

Budget limits, provider routing, auth health, model telemetry, and support for additional providers without deep hardcoding.

0.7

Scalable sandboxed runners

Runner pools, leases, cleanup, isolation controls, safer secret injection, and a Kubernetes or ephemeral-runner path.

0.8

Reliable self-improvement

Incident-to-test pipeline, workflow-doctor maturity, reusable workflow modules, and audited learning from run history.

0.9

Hardening and freeze

Security/privacy review, backup/restore drills, CVM dogfood, docs/API/CLI freeze, and quarantine of unfinished experiments.

1.0

Operator-grade release

Stable workflow-code, runner/sandbox, approval endpoint, budget/provider, install, and upgrade contracts.

Release Refactor Gate

Every major milestone should include a targeted cleanup pass before the tag. For v0.3.0, the cleanup theme is workflow execution contracts and side-effect boundaries.

Fable-led

Architecture and smell map

  • Read the workflow-as-code branch deeply
  • Map duplicated runner, workflow, approval, and provider logic
  • Propose module boundaries and safe slices
  • Review the final diff for architecture quality
Gate-driven

Deterministic release bar

  • No release until focused tests pass
  • Full pnpm test and pnpm build
  • git diff --check
  • Live runner smoke/canary before tag
Contract cleanup

What gets cleaned now

  • Run status and failure taxonomy
  • Replay-safe versus side-effecting steps
  • Runner capability, auth, and budget seams
  • Approval endpoints and stall exclusion
No-go criteria

What blocks v0.3.0

  • Ambiguous replay behavior after commit/push/deploy
  • Stale workflow copies bypassing source changes
  • Approval waits classified as failed or stalled
  • Provider failures hidden as generic failed runs

Questions for Smithers

The goal of the review is to pressure-test the architecture before v0.3.0 hardens around the wrong abstractions.

What should RunYard treat as the stable workflow-as-code contract versus Smithers internals that should remain hidden?
What is the right model for replay safety across checkpoints when a workflow has already committed, pushed, deployed, posted, or mutated shared state?
Should side-effecting workflow steps be modeled as sagas with compensation, explicit finalization nodes, idempotency keys, or some Smithers-native primitive?
How should RunYard pin workflow source/hash/version so resume, fork, and replay do not accidentally run against changed source?
What is the cleanest abstraction for modular approval endpoints such as Telegram, Slack, webhooks, GitHub comments, and in-Hub approvals?
Where should runner sandboxing live: RunYard policy layer, Smithers runtime layer, container/Kubernetes infrastructure, or a combination?
What should provider routing and token budgeting look like if RunYard supports Codex, Fable, Claude, and future providers without a pile of special cases?
How should workflow learning/self-improvement be represented so it is useful and auditable rather than mysterious automatic mutation?
What failure/status taxonomy should exist between Smithers and RunYard so users see blocked, provider-limited, waiting, failed, and unsafe-to-retry correctly?
What architecture would you avoid now because it will become painful once RunYard has many runners, providers, approvals, and workflows?

The ask

Please review whether this direction matches Smithers' intended durability model, where RunYard should rely on Smithers primitives versus add its own control-plane policy, and which abstractions should be locked down before v0.3.0.
